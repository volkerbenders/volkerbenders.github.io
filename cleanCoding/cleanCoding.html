DRY - Dont repeat yourself

Dont implement the same stuff twice.

If you spot code duplications:
  • Use extract to variable
  • Use extract to method
  • ...in general refactor your code to eliminate duplications

    • YAGNI - You aint gonna need it

    Do not write code you don't yet need

    The general rule should be to stick to the necessary minimum amount of code to get your business case going.
    “Don't write code that guesses the future, arrange code so you can adapt to the future when it arrives”

    Write code you don't need

    In case you have to add unused code ensure it does not affect your business logic

    Write code you don't need

    One way could be the feature toggle pattern with implementations like http://www.togglz.org/

    SRP - Single Responsibility Principle

    Each class or method should have a single purpose.

    1. focus!
    2. small methods are easier to test
    3. splitting logic in logical units

    SOC - Separation of Concerns

    Similar to SRP but with a broader scope. A Car class should not implement Engine or Manufacturer logic.

    Trinary Operators

    CAN be very usefull and foster readability:

	result = statement;

	result = "n/a";
}

String result = statement != null ? statement : "n/a";

    Global State

    Set data-state="something" on a slide and "something" will be added as a class to the document element when the slide is open. This lets you apply broader style changes, like switching the page background.

    State Events

    Additionally custom events can be triggered on a per slide basis by binding to the data-state name.
